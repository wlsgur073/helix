# Helix v0.2.0 — release audit sweep (2026-07)

This document records a point-in-time audit of the repository ahead of the v0.2.0 release: a
scan of the full public commit history for accidentally-committed secrets, a license check on
every dependency that ships to users, a review of exactly which files the plugin marketplace
serves, and a baseline of the version string carried at four separate sites in the repository.
It is committed to `docs/release/`, which is tracked and public, so it ships in the release tag
alongside the code it describes.

**Scope.** Repository `wlsgur073/helix` (public on GitHub, full history). Audit performed at
commit `8784565` on branch `release/v0.2.0-prep`, against all locally reachable refs (both local
branches and the `origin` remote-tracking refs — 310 commits total). Every command below was run
against this checkout and its output is reproduced or summarized verbatim.

---

## 1. Public-history scan

Because this repository's history is public, a leaked credential cannot be fixed by deleting the
commit — the fix is rotation/revocation plus an incident note. This scan is written to catch that
case; the outcome below is that no genuine credential was found among the pattern classes this
scan tests, across all reachable history.

**Command 1 — credential-shaped keywords, over all refs:**

```bash
git log --all -p | grep -nE "(api[_-]?key|secret|passwd|password|BEGIN (RSA|OPENSSH)|ghp_[A-Za-z0-9]{20,}|xox[bp]-)" | head -40
```

This pattern matched 813 lines across the full history. All 813 were reviewed (not just the first
40), grouped by what they actually are:

- **The product's own secret-detection subsystem.** Helix ships an egress firewall with legs
  named `secretHeuristic` / `secretEntropy`, so its own source and design comments legitimately
  contain the word "secret" hundreds of times (leg names, regex `kind` labels such as
  `google-api-key` / `pem-private-key`, docstrings). This is the largest category by far.
- **Unit-test fixtures for that same detector**, using well-known, canonical placeholder
  credentials that exist specifically to be recognizable as fake: `AKIAIOSFODNN7EXAMPLE` (AWS's
  own documented example access key), `db_password=Sup3rS3cretValue!` / `api_key=Sup3rS3cretValue123`
  (obviously-synthetic values), and JWT/PEM fixtures with truncated bodies (e.g.
  `-----BEGIN OPENSSH PRIVATE KEY-----\nb3BlbnNzaC...`, `MIIE...`) used only to assert that the
  detector's regex fires.
- **Lexical dictionary data** — `data/semantic-neighbors.json` contains general-English
  synonym/neighbor entries for words including "password," "passwd," and "secret" (e.g.
  `"password":[["authentication",595]]`), which is word-association data, not a credential (see
  §6).
- A few unrelated matches on the substring `api-key` used as an *enum value* for a login-mode
  field (`authMode = 'api-key'`), not a literal key.

No line in this set is a real, usable credential.

**Command 2 — filenames added at any point in history:**

```bash
git log --all --diff-filter=A --name-only | sort -u | grep -iE "\.env|\.pem|\.key|credentials"
```

Zero matches — no `.env`, `.pem`, `.key`, or `credentials`-named file has ever been added to any
reachable branch.

**Command 3 — broader token-shape scan, over all refs** (added for this audit, beyond the
above two): AWS access-key IDs, JWT-shaped strings, and PEM headers, matched anywhere in the
history's diff text:

```bash
git log --all -p | grep -nE "AKIA[A-Z0-9]{16}|eyJ[A-Za-z0-9_-]{20,}|-----BEGIN"
```

31 matches, all accounted for:

- Every `AKIA...` hit is the literal string `AKIAIOSFODNN7EXAMPLE` — again, AWS's own canonical
  documentation placeholder, used in this repo's detector tests and in a `KNOWN_SAFE` exemption
  list so the detector doesn't flag its own test fixtures.
- Two `eyJ...` (JWT-shaped) hits: one is the textbook example JWT (`{"alg":"HS256"}` /
  `{"sub":"1234567890"}`) from a detector unit test, with an obviously-fake signature segment; the
  other is an `integrity: "sha512-..."` field in `package-lock.json` — a normal, public npm
  subresource-integrity hash that coincidentally contains the three characters `eyJ` inside its
  base64 output. Neither is a token.
- Two `-----BEGIN...` hits are the same two truncated PEM fixtures noted under Command 1.

*Methodology note:* re-running the commands in this section against a checkout where this
document itself is committed returns a higher count than reported above, because `git log --all
-p` then also scans this document's own diff — and this section's prose necessarily quotes the
matched substrings (e.g. `AKIAIOSFODNN7EXAMPLE`, the literal text `-----BEGIN`) and, for Command
3, repeats the pattern's own literal `-----BEGIN` alternative in the command text. Measured
directly: once this file entered history, Command 1's count rose from 813 to 828 (+15) and
Command 3's rose from 31 to 36 (+5); every added hit is a line of this document *describing* a
prior finding, not a new one. Command 2 is unaffected — it matches added filenames, and this
document's own filename does not match `\.env|\.pem|\.key|credentials`. Anyone re-running
Commands 1 or 3 later, including after this section is next edited, should expect a count at or
above the figures above and should not treat a higher number, by itself, as a new finding.

**Supplementary check.** A prior pass of this release cycle noted that a filename referencing the
project's ledger-authentication key appears in the tree. Confirmed directly: the only tracked
paths containing "key" are `test/memory/master-key-mint.test.ts` and
`test/verified-read-subkey.test.ts` — test *file names* describing what the tests cover, never
key material, and never in file content.

**Outcome: no genuine credential found among these pattern classes, over all reachable
history.** No blocker-class finding. Nothing to rotate or disclose.

## 2. Dependency licenses

**Runtime dependencies** (the only two packages that ship to users; everything else below is
development-only and does not ship):

```bash
node -e "for (const d of ['@modelcontextprotocol/sdk','zod']) { const p=require(d+'/package.json'); console.log(d, p.version, p.license); }"
```

| Package | Version | License |
|---|---|---|
| `@modelcontextprotocol/sdk` | 1.29.0 | MIT |
| `zod` | 3.25.76 | MIT |

Both are MIT — fully compatible with this repository's own MIT license. No blocker-class finding.

*Methodology note:* the one-liner above prints `undefined undefined` for
`@modelcontextprotocol/sdk` as installed. That package ships both an ESM and a CommonJS build and
its `exports` map has no explicit `./package.json` entry, so Node's module resolution for
`require('@modelcontextprotocol/sdk/package.json')` falls through to a small dual-package-hack
stub at `dist/cjs/package.json` (which contains only `{"type":"commonjs"}`), not the package's
real metadata. The version and license above were confirmed by reading the package's actual root
`package.json` directly. `zod` is unaffected and resolves correctly with the one-liner as written.
Anyone re-running this check on a future dependency bump should read the installed package's root
`package.json` rather than trust a silent `undefined` from this shape of command.

**Development dependencies** (build/test tooling only — never packaged or distributed):

| Package | Version | License |

| `esbuild` | 0.28.0 | MIT |
| `typescript` | 6.0.3 | Apache-2.0 |
| `vitest` | 4.1.8 | MIT |

All permissive and unproblematic for a dev-only role; listed here for completeness, not because


All twelve are ordinary project-root artifacts (license, changelog, security policy, build
config, lockfile). A targeted scan of these files' text for embedded secrets or non-repository
filesystem paths found only the expected documentation prose describing the product's own
secret-detection and trust features — no leaked values, no stray personal paths.

**`.superpowers/` (internal planning/process files) is untracked:**

```bash
git ls-files .superpowers/ | wc -l

```

**`docs/` is tracked only under `docs/release/`.** Nothing else in `docs/` is committed (the
directory is gitignored except for that one path). What `docs/release/` ships, as of this audit:

- `pilot-protocol.md` — the preregistered recall-pilot protocol
- `pilot-manifest.json` / `pilot-oracle-mapping.json` — the frozen pilot manifest and oracle mapping

- `pilot-manifest-amended-1.json` / `pilot-oracle-mapping-amended-1.json` — the amendment's overlay artifacts

`data/` contains exactly one tracked file, `semantic-neighbors.json` (a committed, build-time

**Outcome: nothing private or stray ships.** No blocker-class finding.

## 4. Version-consistency baseline

The release version string is currently duplicated at four sites. This step defines the check so
it can be re-run later in the release cycle, once those four sites are updated together, to
confirm the update landed everywhere at once.

```bash
grep -h '"version"' package.json .claude-plugin/plugin.json | sort -u
python3 -c "import json; print(json.load(open('.claude-plugin/marketplace.json'))['plugins'][0]['version'])"
grep -n "version: '" src/server/helix-server.ts
```

| Site | Current value |
|---|---|
| `package.json` (line 3) | `0.1.0` |
| `.claude-plugin/plugin.json` (line 3) | `0.1.0` |
| `.claude-plugin/marketplace.json` → `plugins[0].version` (line 9) | `0.1.0` |
| `src/server/helix-server.ts` (line 17, `McpServer` construction) | `0.1.0` |

**Baseline: consistent-but-old.** All four sites agree with each other, and all four still say
`0.1.0` — expected at this point in the cycle, since the version bump has not happened yet. A
later step in this same release cycle updates all four sites to `0.2.0` together; re-running the
three commands above afterward should show `0.2.0` at every site with no stragglers.

## 5. Semantic-neighbors "kim" token note

`data/semantic-neighbors.json` (see §3) is a build-time, general-English word-association table
used to broaden recall queries with synonyms. It contains an entry for the token `kim`:

```
"kim":[["kimberly",673], ...]
```

This is lexical content, not a personal file path or an identifier for any specific individual:
`kim` here is the common English given-name token, linked to its longer form `kimberly` the same
way the table links any other short name to related forms. Nothing in this file is
project-specific or derived from this repository's own commit history or contributors.

## 7. Deferred-with-note items

Items noticed during this sweep that are intentionally not acted on in this document, with the
reason each is safe to defer:

1. **Version mismatch (four sites, all `0.1.0`; §4).** Deferred to a later step in this same
   release cycle, which updates all four together. Not a defect on its own — it is the expected
   pre-bump baseline, recorded so the post-bump state has something exact to diff against.
2. **`@modelcontextprotocol/sdk` package-metadata lookup (§2).** The literal single-line Node
   check silently returns `undefined` for this one package due to its dual CJS/ESM packaging. Not
   a licensing risk (the real version/license were confirmed by reading the package's root
   `package.json`), but noted so a future run of this check isn't misread as a missing-license
   finding for a package that in fact has a clear MIT license.
3. **Broader trust-model residual risks are out of this document's scope.** This sweep covers
   public history, dependency licenses, packaged files, and version consistency only. The memory
   system's own trust model has separate, already-published residual-risk notes (e.g., what an
   adversary with local filesystem read access can and cannot do) documented in `SECURITY.md`'s
   own disclosures; this audit does not re-derive or duplicate them.

No item in this list is a blocker-class finding (real secrets, a license-incompatible runtime
dependency, or data-loss risk).
